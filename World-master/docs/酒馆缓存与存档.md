# PR：酒馆缓存与存档（跨设备同步 + 防丢失存档）

> 给「🌍 世界引擎」加入「把按聊天隔离的存档存进酒馆聊天本身」的能力：
> 数据随聊天文件保存到 SillyTavern 服务器，从而**跨设备同步**，并提供**命名存档 / 滚动自动备份**防丢失。
>
> 版本：`2.1.0` → `2.2.0`

---

## 1. 背景与动机

现状：本扩展所有存档都走 [`world-engine-store.js`](../world-engine-store.js)，落在 **IndexedDB**（不可用时回退 localStorage）。
IndexedDB 是**按「设备 + 浏览器」隔离**的本地存储——

- 换一台电脑 / 换一个浏览器 / 清理浏览器数据，世界状态就没了，哪怕聊天本身还在酒馆服务器上。
- 用户的真实诉求：「**对应聊天里的缓存。就算换了设备，只要还在酒馆，缓存还在。**」

而 SillyTavern 自身有一个天然就「跟着聊天走、存在服务器上、跨设备同步」的容器：**`chat_metadata`**。
它会被序列化进聊天文件的文件头，随聊天一起保存到酒馆服务器，再随聊天一起被任何设备载入。

本 PR 就是把「本扩展、当前聊天」的存档**镜像进 `chat_metadata`**，从而实现跨设备同步与防丢失。

---

## 2. SillyTavern 机制确认（为什么这样能跨设备）

读 SillyTavern `release` 分支源码确认（`public/script.js` / `public/scripts/st-context.js` / `public/scripts/extensions.js`）：

| 时机 | 行为 | 源码位置 |
|------|------|----------|
| 打开聊天 | `chat_metadata = chatHeader.chat_metadata`（从聊天**文件头**载入）**先于** `CHAT_LOADED` 事件触发 | `script.js` 载入聊天处（`chat_metadata = chatHeader?.chat_metadata ?? {}` → 紧接 `eventSource.emit(CHAT_LOADED, ...)`） |
| 保存聊天 | `saveChat` 把 `chat_metadata` 写进聊天**文件头** `{ chat_metadata: metadata, ... }`，POST 到服务器 | `script.js` `saveChat()` 组装 `chatHeader` 处 |
| 读 / 写 API | `getContext()` 暴露 `chatMetadata`、`updateChatMetadata(values, reset)`、`saveMetadataDebounced()`、`saveMetadata()`、`saveChat` | `st-context.js` |
| 持久化去抖 | `saveMetadataDebounced()` 去抖且**对群聊安全**（切了 group/character 会自动放弃保存） | `extensions.js` `saveMetadataDebounced()` |

两条关键结论：

1. **`CHAT_LOADED` 触发时 `chat_metadata` 已就位** → 本扩展现有的 `onChatLoaded` 里可以直接读到同步过来的存档。
2. **`saveChat` 把 `chat_metadata` 写进聊天文件（服务器侧）** → 这正是「跨设备」的载体。

> ⚠️ `chat_metadata` 在切聊天时是被**整体替换**（重新赋值）的，缓存旧引用会写错聊天。
> 因此本模块**每次都用 `SillyTavern.getContext()` 取最新引用**。

## 3. 方案概述

在 `chat_metadata.world_engine` 命名空间下放两样东西，二者都随聊天跨设备：

1. **实时同步（`live`）** —— 开关 `syncToChat`，**默认关闭**。
   开启后，工作区状态持续镜像进聊天；换设备打开同一聊天即可续上进度。
   冲突按「**较新修订号胜出**」（Lamport 计数器，见 §6）。

2. **聊天存档（`snapshots`）** —— 防丢失。
   - **命名手动存档**：随时点「新建命名存档」起名保存。
   - **滚动自动备份**：开关 `autoBackup`，**默认关闭**；开启后每当轮次推进自动存一条，保留最近 3 条。
   - 每条存档支持：**恢复 / 重命名 / 导出(JSON) / 删除**；并支持**导入**外部导出的存档文件。

> 两个开关都**默认关闭**：不打开就**完全不写聊天文件**，对现有用户零行为改变（命名存档是显式点击，不算自动写入）。

## 5. 数据结构

`chat_metadata.world_engine`：

```jsonc
{
  "v": 1,                      // schema 版本
  "live": {                    // 实时同步槽（仅 syncToChat 用到；否则为 null）
    "rev": 12,                 // Lamport 修订号
    "updatedAt": 1700000000000,// 调试用
    "chatId": "xxx",
    "data": { /* slots */ }
  },
  "snapshots": [               // 存档列表（新→旧）
    {
      "id": "s...",
      "name": "决战前夜",
      "auto": false,           // true=自动备份, false=手动/导入
      "round": 23,
      "createdAt": 1700000000000,
      "chatId": "xxx",
      "data": { /* slots */ }
    }
  ]
}
```

**slots** = 本扩展「按聊天隔离」的 5 个键，**原样存字符串**，按槽名存放（不存完整 key，便于聊天改名/复制后重新落到当前 chatId）：

| 槽名 | 对应 store 键 | 来源 |
|------|---------------|------|
| `state` | `world_engine_<id>` | core 主状态 |
| `checkpoint` | `world_engine_<id>_checkpoint` | core 存档点 |
| `fingerprint` | `world_engine_<id>_fingerprint` | core 指纹 |
| `anchorLayer` | `world_engine_<id>_anchorLayer` | core 旧版锚点 |
| `worldbook` | `world_engine_worldbook_selection_<id>` | 后台推演世界书选择 |

瘦身：打包 `state`/`checkpoint` 时剥离纯调试、可由 `ensureArrays` 重建的字段（`lastInjection`、`lastEvolveResult`），减小聊天文件体积。

> **绝不写入** 全局设置键 `world_engine_settings`（内含 **API Key**）——见 §7。

## 6. 冲突解决：Lamport 修订号（「较新胜出」）

跨设备「谁更新」不能依赖时间戳（设备时钟可能不一致）。采用 **Lamport 计数器**：

- 每台设备本地记 `world_engine_<id>_syncrev`（仅本地、不进聊天、不触发同步）。
- 推送 live 时：`新 rev = max(本地 rev, 聊天里现有 rev) + 1`。
- 打开聊天时：
  - 聊天 `rev` > 本地 → **采用云端**（安装进 store，本地 rev = 云端 rev）。
  - 聊天 `rev` < 本地 → **本地较新**，推上去收敛。
  - 相等 → 已同步。

效果（已测，见 §10 场景1）：A 推到第10轮→B 打开续到第10轮→B 推到第11轮→A 打开续到第11轮，单调收敛。

**数据安全护栏**：本地**没有任何存档**时（正是「丢了数据的设备」），**绝不**用空内容推送/覆盖云端 live——
否则一个空设备先开同步就会把别处的真实存档清空（精确安装会删 slot）。已测（场景2）。

## 7. 安全与隐私

- **只镜像 5 个「按聊天隔离」的存档键**，**绝不**把 `world_engine_settings` 写进聊天文件。
  该设置里含 **API URL / API Key**；聊天文件可能被分享、导出，泄露密钥后果严重。这是刻意的边界。
- 写 `chat_metadata` 只动 `world_engine` 这一个键（`updateChatMetadata` 顶层浅合并），
  保留 ST 的 `integrity`、作者注释等其他扩展写入的数据。

## 8. 边界情况

| 情况 | 处理 |
|------|------|
| 无聊天 / `chatId === 'default'` | 跳过一切 chat_metadata 读写（`chatUsable()` 守卫） |
| 旧版酒馆无 `updateChatMetadata`/`saveMetadataDebounced` | 逐级回退到 `saveMetadata`/`saveChat`；都没有则功能优雅降级（不崩） |
| 切聊天竞态 | 始终用 fresh `getContext()`；推送用 `ctx.chatId`；`saveMetadataDebounced` 本身切聊天会放弃保存 |
| 安装存档回弹 | 安装期间挂起同步槽 `_suspend`，避免「写回 store → 又被当成新写入推回聊天」 |
| 恢复历史存档后楼层错乱 | 把 `chatLayer`/`fingerprint` 归一化到当前对话层数（与既有「数据导入」逻辑一致），避免被误判为重 roll |
| 恢复误操作 | 恢复前自动生成一条「恢复前自动备份」，可再恢复回来 |
| 聊天文件体积 | 自动备份滚动保留 3 条、手动 20 条；命名空间超约 1MB 时 `console.warn` 软告警 |
| 群聊 | `chatId` 为群聊 chat_id，`saveMetadataDebounced` 群聊安全；应可用（建议重点回归测试） |
| 30s 面板自动刷新 | 设置页不被自动刷新清空（既有逻辑 `auto && view==='settings'` 直接 return） |

## 9. 用户怎么用

设置页新增分区 **「酒馆缓存与存档」**：

- ☐ **跨设备实时同步（存进当前聊天）** —— 开启后本聊天世界状态持续写入聊天文件并跨设备同步。
- ☐ **自动滚动备份（每当轮次推进存一条，保留最近 3 条）**
- 状态行：`实时同步已开启 · 本地修订 12 / 云端 12 · 共 5 条存档`
- 按钮：**新建命名存档**、**导入存档**
- 列表：每条存档显示 `[手动/自动] 名称 · 第N轮 · 时间`，行内操作 **恢复 / 重命名 / 导出 / 删除**

## 10. 测试

写了离线逻辑测试（用 `vm` 沙箱加载**真实**的 `world-engine-chatcache.js`，mock `SillyTavern`/`store`/`core`，
两台「设备」共用同一个 `chat_metadata` 对象模拟服务器同步），**18/18 全过**：

1. **场景1 实时同步较新胜出**：A↔B 交替推进，单调收敛；live 已剥离 `lastInjection`。
2. **场景2 空内容护栏**：空设备开同步不会清空云端真实存档，反而从云端恢复。
3. **场景3 存档 CRUD**：创建/列表/重命名/恢复/导出/导入/删除全部正确；恢复前自动备份生成。
4. **场景4 滚动备份**：推进 6 轮后自动备份只剩最近 3 条，最新为第 6 轮。

另：5 个改动的 JS 文件均通过 `node --check` 语法校验。

> 测试脚本为离线验证用途，未纳入仓库（项目本身无测试框架）。如需保留可放到 `tests/` 下。

**建议人工回归**：
- 单角色聊天：开同步 → 推演几轮 → 换浏览器/隐身窗口打开同一聊天，确认续上。
- 命名存档 → 推演 → 恢复，确认面板与注入都回到存档点。
- 群聊场景；旧版酒馆（无 `updateChatMetadata`）的优雅降级。

## 11. 已知限制与后续

- **最后写入者胜**：两台设备**同时**编辑同一聊天再保存，后保存者覆盖前者（SillyTavern 聊天文件模型本身如此）。世界引擎是单人轮流使用，影响极小。
- **离线**：离线期间 `saveMetadataDebounced` 的服务器保存可能失败，待下次聊天保存时落盘（尽力而为）。
- 后续可考虑：自动备份条数 / 触发策略做成可配置；面板内「从酒馆强制拉取」按钮；命名空间体积硬上限。
