# PR：批量「重填世界推演」（分批回填到指定楼层）

> 给「🌍 世界引擎」加入**从第 1 个 AI 楼层分批把世界状态重新推演到指定楼层**的能力：
> 装扩展前已有大量对话、或想推倒重来时，能一键按批回填，让世界状态补齐且连贯。
>
> 版本：`2.3.0` → `2.3.1`
> 目标分支：`cloud`（已含「酒馆缓存与存档 v2.2.0」+「世界书蓝绿灯触发 v2.3.0」，本 PR 叠加其上）

---

## 1. 背景与动机

现状：世界引擎只能**逐轮**推演——每收到一条 AI 回复，推进世界一步（[`world-engine.js`](../world-engine.js) `onMessageReceived` → `performEvolution`）。

痛点：

- 用户常在**已有几十层对话之后**才安装/启用扩展，此时世界状态是**空白**的，与已经发生的剧情严重脱节。
- 或者用户调整了规则/世界书，想让世界**推倒重来**，但没有「从头重算」的入口。

本 PR 提供「**重填世界推演**」：从第 1 个 AI 楼层开始，**分批**（每 N 个 AI 楼层一批）把世界推到指定楼层。30 个 AI 楼层、每 5 层一批 → 调用 6 次推演，模拟「世界逐段向前推进」。

---

## 2. 设计决策（与用户敲定）

1. **对话范围 = 仅本批 N 层**（非累积）。
   第 1 批只喂第 1–5 层对话，第 2 批只喂第 6–10 层……每批喂给 API 的 token **恒定可控**，不随批次暴涨。
2. **但世界状态逐批累积**。
   第 k 批在第 k−1 批的**推演结果之上**继续推（每批结束 `state` 已落盘，下一批重新读取），保证世界连贯——第 2 批能看到第 1 批产生的事件链、风声、势力变化。
3. **起点 = 清空重来**。
   回填前**丢弃当前世界状态与存档点**，从空白世界 + 第 1 个 AI 楼层起推。符合「重填」字面含义。
   - 回填前**自动存一份备份快照**（`回填前自动备份`），万一不满意可恢复。
   - 开始前**弹确认框**明确告知「会清空重来」。
4. **每批独立、可配置的重试次数**。
   某批推演失败（API 报错/解析失败）时，按设定上限**独立重试该批**；重试用尽才**整体中止**（不在坏状态上继续叠加）。
5. UI 可配置：**每批 AI 楼层数**、**结束楼层**（0 = 推到最后）、**每批重试次数**。

> 本功能是**显式手动触发**（点按钮 + 确认），不改变任何自动推演行为，对现有用户零影响。

---

## 3. 复用既有单批推演链

关键：**不重写推演逻辑**，批量回填 = 串行调用 N 次现成的 `evolve(forward)`。

- 单批推演 `evolution.evolve(state, userMsg, aiMsg, { mode:'forward', dialogueText })`（[`world-engine-evolution.js`](../world-engine-evolution.js)）内部已做齐：
  - `mode:'forward'` → 视为新轮次，`state.round++`、滚动存档点、`saveStateWithLayer(state)` 把楼层收敛到当前真实层。
  - 返回 `true/false`；失败时已 `Object.assign(state, backup)` **回滚**，可安全重试。
- `callEvolutionAPI` 把 `dialogueText` 直接作为「## 近期对话」喂入 prompt，所以每批只需构造该批对话文本。

> 因此回填结束后 `round` 与 `chatLayer` **天然正确**，无需额外归一化。

---

## 4. 回填引擎：`backfillEvolve(opts)`（`world-engine-evolution.js`）

新增并导出 `async function backfillEvolve(opts)`，`opts = { batchSize, retries, endLayer, onProgress }`。

### 4.1 计算 AI 楼层索引表
遍历 `SillyTavern.getContext().chat`，收集所有 `!is_user && mes.trim()` 的下标，得 `aiIdx = [i0, i1, ...]`（判据与自动推演 `onMessageReceived` 一致：排除 user 楼层、空消息）。`aiIdx.length` = 有效 AI 楼层数。

### 4.2 结束楼层夹紧
`endLayer` 缺省/为 0/超过实际层数 → 取 `aiIdx.length`（推到最后）。

### 4.3 切批（含不整除处理）
前 `endLayer` 个 AI 楼层按 `batchSize` 切片，**末批吸收余数**——若最后一批不足 `batchSize` 则并入上一批：

| 总层 / 每批 | 批次（每批层数） |
|------------|------------------|
| 30 / 5 | 5,5,5,5,5,5（整除，6 批） |
| 30 / 7 | 7,7,7,**9**（末批吸收余数） |
| 14 / 7 | 7,7（恰好整除，不误并） |
| 10 / 20 | 10（batchSize ≥ 总数 → 1 批） |

### 4.4 清空重来
`core.clearState()` + `core.clearCheckpoint()`，让第 1 批从空白世界起推（强制 `mode:'forward'`，绕过 `isNewRound()` 指纹判断）。

### 4.5 逐批循环
对每批 `[pStart..pEnd]`（AI 楼层序号）：
- **本批对话文本**：取 chat 中「上一个 AI 楼层之后」到「本批最后一个 AI 楼层」的切片（含夹在中间的 user 楼层），用与 `performEvolution` 一致的格式：`'用户'/'AI'：' + core.filterDialogue(...)`，`filter(line.length>3)`，`join('\n')`。
- `aiMsg` = 本批最后一个 AI 楼层的 `mes`。
- **重试循环** `for (attempt = 0; attempt <= retries; attempt++)`：每次 `core.loadState()` 重读（evolve 已落盘）→ `evolve(state, '', aiMsg, { mode:'forward', dialogueText })`；成功 break；失败且未中止则重试；用尽则**整体中止**。
- 每批成功后 `WORLD_ENGINE_LEDGER.recordChanges(state)`（与单轮路径一致）。
- 通过 `onProgress({ phase, batch, totalBatches, layerFrom, layerTo, attempt, ok, round })` 回调上报进度供 UI 显示。

### 4.6 中止与守卫
- 新增模块级 `_backfillRunning` / `_backfillAborted`。
- `abort()` 同时拉起 `_backfillAborted`（让回填在批间隙优雅停止）并 `_abortController.abort()`（中断进行中的单批请求）。
- `isRunning()` 返回 `_isRunning || _backfillRunning`——回填期间，事件触发的自动推演被既有 `evolution.isRunning()` 守卫挡住，不会插入。

返回 `{ done, totalBatches, completedBatches, failedAt?, reason? }`。

## 5. 触发与编排：`runBackfill()`（`world-engine-ui.js`）

新增模块级 `async function runBackfill()`：

1. 读设置 `backfillBatchSize / backfillRetries / backfillEndLayer`，统计当前 AI 楼层数，算出有效结束楼层与批次数。
2. `confirm()` 弹确认：「将清空当前世界状态，从第 1 个 AI 楼层重新推演到第 N 层，共约 M 批、每批最多重试 R 次。开始前会自动存一份备份快照。确定推倒重来？」
3. 确认后先 `window.WORLD_ENGINE_CHATCACHE?.createSnapshot('回填前自动备份')`（失败仅 `console.warn`，不阻断）。
4. 设 `isEvolving`、`setEvolvingUI(true,'state')`，调 `evolution.backfillEvolve({ batchSize, retries, endLayer, onProgress })`。
   `onProgress` 把「第 i/M 批…」写入设置页状态行 `#we-backfill-status` 与悬浮球外部状态。
5. 结束按 `result.reason` 分支提示（完成 / 中止 / 某批失败 / 无 AI 楼层），并 `applyInjection()` 让正文注入跟上新状态。
6. 防重入：复用 `isEvolving` 与 `evolution.isRunning()` 守卫。

---

## 6. 设置 UI（`world-engine-ui.js` + `world-engine-api.js`）

`renderSettingsForm` 新增独立分区 **「批量重填世界推演」**（置于「推演模式」之后）：
- 三个数字框：`每批 AI 楼层数`（min 1）、`结束楼层（0=全部）`（min 0）、`每批重试次数`（min 0），onchange 即时落盘（既有 `persist` 模式）。
- 两个按钮：`▶ 开始重填世界推演`（主按钮 → `runBackfill()`）、`■ 停止`（→ `evolution.abort()`）。
- 一行说明 + 状态行。

`world-engine-api.js` 默认设置新增：
```js
backfillBatchSize: 5,    // 每批 AI 楼层数
backfillRetries: 2,      // 每批独立重试次数
backfillEndLayer: 0,     // 结束 AI 楼层（0 = 推到最后）
```

## 8. 边界情况

| 情况 | 处理 |
|------|------|
| 无聊天 / AI 楼层数为 0 | 提示「当前聊天没有可推演的 AI 楼层」，不执行 |
| `endLayer` > 实际 AI 层数 | 夹紧到实际层数 |
| `batchSize` ≥ 总层数 | 退化为 1 批（仍正常工作） |
| 不整除 | 末批吸收余数（30/7 → 7/7/7/9），不产生空批 |
| 某批重试用尽仍失败 | 记入 `failedAt`，**中止后续**（不在坏状态上叠加），提示停在第几批 |
| 用户点停止 | `abort()` → 当前批 `evolve` 抛 `AbortError` / 批间隙检测到 `_backfillAborted` → 优雅终止，已完成批保留 |
| chatcache 不可用（无同步/不可写） | 备份快照静默跳过（warn），回填照常进行 |
| 回填期间自动推演事件触发 | `isRunning()` 含 `_backfillRunning`，既有守卫阻止插入 |
| 30s 设置页自动刷新 | `refresh(auto)` 在设置页直接 return（既有护栏），不清空输入框/状态行 |

## 9. 测试

无测试框架，按项目既有方式：

1. **语法**：改动的 3 个 JS 均通过 `node --check`；`manifest.json` 通过 `JSON.parse` 校验。
2. **切批算法离线测试**（纯算法复刻，14 项全过）：整除（30/5→6 批均匀）、不整除（30/7→7/7/7/9）、末批吸收余数、`endLayer` 夹紧、`endLayer` 小于总数、整除不误并（14/7→7/7）、单层、0 层、`batchSize=1`、`batchSize ≥ 总数`。

**建议人工回归**：
- 一个有 ≥10 个 AI 楼层的聊天，设每批 3 层、重试 1 → 点开始 → 确认 → 观察 F12 逐批日志、面板 `round` 递增、世界状态连贯。
- 中途点「停止」，确认优雅终止、已完成批保留。
- 「酒馆缓存与存档」列表里出现「回填前自动备份」。
- 不整除（如 10 层每 3 → 3/3/4）确认末批含正确楼层。

## 10. 已知限制与后续

- **每批是独立 API 调用**：回填 30 层每 5 批 = 6 次 API 往返，耗时与费用与正常推演 6 轮相当。大聊天回填请预留时间。
- **失败即停**：某批用尽重试仍失败则整体中止（而非跳过续推），避免世界在断裂状态上继续叠加。已完成批已落盘，修好后可调小结束楼层从断点附近续填（后续可做「断点续填」）。
- 后续可考虑：累积喂历史模式（可选）、断点续填、回填进度条 UI、每批间可配置延迟（规避 API 限流）。
