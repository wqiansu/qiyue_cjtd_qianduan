# 升级后内存膨胀 / 批量回填切聊天存档污染 修复（v2.3.3）

## 背景

用户反馈「版本升级后酒馆会炸内存、浏览器崩溃」。经 11 个模块并行泄漏扫描 + 6 处 top 嫌疑独立交叉验证，锁定 **3 条根因，全部在 v2.2.0 周期**（「酒馆缓存与存档(chatcache)」+「批量重填(backfill)」同期引入），属"随新功能引入的回归"，与用户反馈时序吻合度高。

另：交叉验证推翻了 3 个看似吓人但实际不成立的指控（不在本 PR 修复范围）——
- 主入口 4 个 `eventSource.on` 无翻倍累积（`__WORLD_ENGINE_LOADED__` 守卫无重置路径 + 页面 reload 同步销毁旧监听）
- 30s `setInterval` 不产孤儿 DOM（`refresh()` 面板隐藏时早退）
- 四数组（factions/worldTrends/events/winds）有硬上限（core.js 15/4/16/12 pop），非无界增长

## 三根因与修复

### 🔴 根因二（Critical · 毁存档）：批量回填切聊天数据污染

`backfillEvolve` 在开头一次性捕获 `chat = ctx.chat`（A 聊天的消息数组引用），但循环内每批 `core.loadState()`/`saveStateWithLayer()`/`saveFingerprint()` 都用**动态** `getChatId()`。每批 `await api.callApi` 有最长 8 秒空窗，用户在此期间切到聊天 B，整段回填就变成「读 B 的 state + 喂 A 的对话文本 + 写回 B 的存档」——且开头已 `core.clearState()` 把 A 清空。后果三连：

- **A 存档丢失**：开头 `clearState()`/`clearCheckpoint()` 清的是 A 的存档，切走后再不写回 A。
- **B 被污染**：A 的对话内容被推演成世界状态写进 B。
- **B 自动推演静默失效**：`_backfillRunning=true` 使 `isRunning()` 恒 true，B 的 `GENERATION_ENDED` 触发的自动推演被持续跳过，直到回填结束。

主入口 `onChatLoaded` 原先不调用 `evolution.abort()`，无自动刹车——回填会一直跑到结束或某批失败。

**修复**（[world-engine-evolution.js](../world-engine-evolution.js) `backfillEvolve` + [world-engine.js](../world-engine.js) `onChatLoaded`）：
1. `backfillEvolve` 捕获 chat 时同时记 `const startChatId = core.getChatId()`。
2. 批循环顶部、每次 retry attempt 开头各加 `if (core.getChatId() !== startChatId) return { reason: 'chat-changed' }`——一旦切聊天立即返回，不再往 B 写。两个检查点覆盖「批间隙」和「await API 边界」。
3. `onChatLoaded` 顶部加 `if (evolution.isRunning()) evolution.abort()`——切聊天时立即中断在途 API 请求（不必等到下一个批边界），复用既有 `abort()`，不改它。

> 说明：A 在 backfill 开头被 `clearState` 是「清空重来」的固有设计；切聊天中止后 A 存档已部分丢失属已知后果（用户可从「回填前自动备份」快照恢复）。本修复的核心是**阻止污染 B** 并立即停车，不试图复活已删的 A。

### 🟠 根因一（High · 每次推演都全量写聊天）：live 无内容去重

`pushLiveNow` 的 `rev = ... + 1` 无条件 +1，从不比较新 `packChat` 与 `ns.live.data` 是否相同。只要本地有任一 slot 有值，`packChat` 必非空 → 必返回非 null rev → `runTick` 必 `writeNamespace` → `saveMetadataDebounced` 全量写聊天文件。批量回填 N 批 = N 次全量聊天写入 + rev 单调暴涨（同批内 state+checkpoint+fingerprint 三次 setItem 被去抖合并为 1 个 tick，粒度是「每批/每步」）。

**修复**（[world-engine-chatcache.js](../world-engine-chatcache.js)）：
- 新增 `sameData(a,b)`：逐 slot 字符串 `===` 比对（slot 值本就是字符串，无需 `JSON.stringify` 双向序列化）。
- `pushLiveNow(nsArg, force)` 加 `force` 形参：`!force && ns.live.chatId===id && sameData(ns.live.data, data)` 时返回当前 `ns.live.rev` 不 +1、不更新 live、不写盘。
- `runTick` 记 `prevRev`，仅 `r !== prevRev` 才 `changed=true`——内容不变就不触发 `writeNamespace`。
- `restoreSnapshot` 显式 `pushLiveNow(ns, true)` 强制推（恢复后即使内容相同也要把 live 指向恢复后的状态，避免下次 `onChatLoaded` 被云端旧值覆盖）。

### 🟠 根因三（Partial · CPU/IO 风暴）：chat_metadata 24× 状态序列化

`ns` 常驻 live(1) + 自动备份(3) + 手动(20) ≈ **24 份完整状态副本**塞在 `chat_metadata.world_engine`。每次 tick `writeNamespace` 把整份 24× ns `JSON.stringify` **仅为取 `.length` 做软告警**，字符串随即丢弃；`addAutoBackupIfAdvanced` 又 `JSON.stringify` 比对两次。本身不致浏览器 OOM（MB 级，SIZE_WARN_BYTES=1MB 阈值佐证作者预期 MB 级），但叠加根因一的高频触发，是 CPU 阻塞 + 聊天文件 I/O 风暴的主源，长时间表现为"卡死、疑似内存爆炸"。

**修复**（均在 [world-engine-chatcache.js](../world-engine-chatcache.js)，最小改）：
- 新增 `nsSize(ns)`：直接累加各 `live.data`/`snapshot.data` 的 slot 字符串长度（data 本就是字符串）+ 固定元数据开销，替代 `JSON.stringify(ns).length`。
- `addAutoBackupIfAdvanced` 用 `sameData(newestAuto.data, packed)` 替代双 `JSON.stringify` 比对。
- `ensureNamespace` 末尾调一次 `pruneSnapshots(ns)`：升级后旧条数可能超过现 limit，打开聊天即时收敛（堵"prune 仅在下一次 addSnapshot 才触发"的迁移缺口）。
- `onChatLoaded` 开头 `clearTimeout(_tickTimer)`：丢弃上一个聊天遗留的 pending tick，防其在 B 上下文意外写盘/生成自动备份（与根因一同源）。

### 附加（同症状族，低风险）：evolve 复位入 finally

`evolve` 的 try/catch 把 `_abortController=null; _isRunning=false;` 复位指令分别写在 try 末尾和 catch 末尾。若 catch 里的恢复语句 `Object.assign(state, backup); core.saveState(state)` 在内存压力下（IDB 写失败高发）抛错，会跳出 catch，复位指令永不执行 → `_isRunning` 永久 true → 后续所有 `evolve` 被 `isRunning()` 守卫 `return false` 静默失效，表现为「推演再也不工作了」（除非刷新页面）。

**修复**：把复位指令移入 `finally` 块，恢复语句用内层 try/catch 吞错。仅调整复位位置与错误兜底，不动推演算法。属「升级后内存压力下偶发推演卡死」的同源修复。

## 复现脚本（建议 reviewers 实跑）

```
1. 准备一个含 100+ AI 楼层的长聊天（升级前旧存档）
2. 升级到 v2.3.0+，刷新页面
3. 开 DevTools Console，过滤 [世界引擎]
4. 在该聊天执行「批量重填」batchSize=1, endLayer=0（全推）
5. 在第 3 批执行中切到另一个干净聊天 B
6. 等待几秒
预期（修复后）：
  - Console 立即打印「🛑 检测到切聊天，中止批量回填（start A → now B）」
  - 返回 reason 'chat-changed'
  - B 的世界书/状态不含 A 的对话内容（未被污染）
  - B 的自动推演恢复正常（_backfillRunning 已复位）
对比（修复前）：
  - 回填继续跑完，A 存档被清空、B 被注入 A 内容、B 自动推演持续被跳过
```

根因一/三的验证：连续手动 evolve 多步，若 state 无实质变化，`saveMetadataDebounced` 不应再被频繁触发（可在 chatcache 调用点临时埋点统计）。
